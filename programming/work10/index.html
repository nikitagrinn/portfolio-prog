    Создание сайта-портфолио (MkDocs + GitHub Actions)¶

    Цель работы¶

    Реализовать документацию к лабораторным работам и развернуть её в виде статического сайта-портфолио в сети Интернет.

    Задачи:

    1. Изучить работу с генератором статических сайтов MkDocs.
    2. Настроить тему оформления Material for MkDocs.
    3. Создать структуру документации с использованием языка разметки Markdown.
    4. Настроить CI/CD (непрерывную интеграцию и доставку) через GitHub Actions для автоматической сборки и публикации сайта на GitHub Pages.

    Что такое MkDocs и как он работает¶

    MkDocs — это быстрый и простой генератор статических сайтов, специально разработанный для создания проектной документации.

    Принцип работы:

    1. Исходные файлы пишутся в простом текстовом формате Markdown (.md).
    2. Все настройки проекта (название, меню, тема) хранятся в одном YAML-файле mkdocs.yml.
    3. При запуске команды сборки (mkdocs build), генератор читает Markdown-файлы и конвертирует их в красивые HTML-страницы со связанной навигацией, стилями и поиском.

    В данном проекте используется расширенная тема Material for MkDocs, которая даёт современный дизайн (похожий на Material Design от Google), адаптивность для мобильных устройств, поддержку тёмной темы и удобный поиск.

    Архитектура и структура проекта¶

    Репозиторий разделён на две основные части: конфигурация сайта и исходный код с контентом.

    portfolio-prog-main/
│
├── mkdocs.yml                  # Главный конфигурационный файл сайта
├── README.md                   # Описание проекта для GitHub
├── requirements.txt            # Список зависимостей Python (mkdocs, material)
│
├── .github/workflows/
│   └── deploy.yml              # Скрипт автоматического деплоя (GitHub Actions)
│
└── docs/                       # Папка со всем содержимым сайта
    ├── index.md                # Главная страница
    ├── about.md                # Страница "Обо мне"
    ├── contacts.md             # Страница "Контакты"
    │
    └── programming/            # Раздел с лабораторными работами
        ├── work1.md            # Отчёт по Теме 1
        ├── work2.md            # Отчёт по Теме 2
        ├── ...
        │
        ├── lab1/               # Исходный код к Теме 1
        │   └── script.py       
        ├── lab2/               # Исходный код к Теме 2
        │   ├── main.py         
        │   └── test_app.py     
        └── images/             # Скриншоты и изображения

    Конфигурация сайта (mkdocs.yml)¶

    В файле mkdocs.yml задаются основные параметры сайта. Вот ключевые блоки из нашего проекта:

    1. Настройка темы¶

    Мы используем тему material и настраиваем цветовую палитру (переключатель светлой и тёмной темы):

    theme:
  name: material
  language: ru

  palette:
    # Светлая тема
    - scheme: default
      primary: indigo
      accent: indigo
      toggle:
        icon: material/weather-night
        name: Темная тема

    # Темная тема
    - scheme: slate
      primary: indigo
      accent: indigo
      toggle:
        icon: material/weather-sunny
        name: Светлая тема

    2. Навигация (меню)¶

    Блок nav строго задаёт структуру меню слева (и сверху). Каждый пункт меню ссылается на соответствующий .md файл внутри папки docs/:

    nav:
  - Главная: index.md
  - Обо мне: about.md
  - Программирование:
      - Тема 1. Two Sum: programming/work1.md
      - Тема 2. Вычисление деления: programming/work2.md
      # ... и так далее
  - Контакты: contacts.md

    3. Расширения Markdown¶

    Чтобы в отчётах корректно отображались блоки кода с подсветкой синтаксиса, таблицы и сноски, подключены плагины pymdownx:

    markdown_extensions:
  - admonition
  - tables
  - pymdownx.highlight      # Подсветка синтаксиса
  - pymdownx.superfences    # Вложенные блоки кода
  - md_in_html              # Использование Markdown внутри HTML-тегов

    Встраивание PDF-документов (Презентаций)¶

    В проекте (например, в Теме 6) потребовалось добавить презентацию (PDF) так, чтобы она отображалась прямо на странице сайта, а не скачивалась как отдельный файл.

    В Markdown нет стандартного синтаксиса для предпросмотра PDF, поэтому была использована связка Markdown и HTML-тега <iframe>:

    <iframe 
  src="../Pattern-Strategiya-Strategy.pdf"
  width="100%"
  height="850px"
  style="border: none; border-radius: 8px;"
>
  Ваш браузер не поддерживает встроенный просмотр PDF.
  <a href="../Pattern-Strategiya-Strategy.pdf">Скачать PDF</a>
</iframe>

    Этот подход позволяет: 1. Автоматически масштабировать презентацию под ширину экрана (width="100%"). 2. Давать красивый fallback (запасной вариант со ссылкой на скачивание), если человек зашёл со старого мобильного устройства, не поддерживающего встроенный просмотр PDF.

    Запуск сайта на локальном компьютере (Localhost)¶

    В процессе разработки документации важно иметь возможность видеть изменения в реальном времени до их отправки на сервер.

    Для локального запуска используется встроенный сервер MkDocs.

    Шаги для запуска:

    1. Установка зависимостей: В терминале в папке проекта необходимо установить библиотеку и тему с помощью pip: 
      pip install -r requirements.txt
    2. Запуск сервера: В корневой папке проекта (там, где лежит файл mkdocs.yml) выполняется команда: 
      mkdocs serve
    3. Просмотр результата: MkDocs соберёт проект и запустит локальный веб-сервер. В терминале появится ссылка вида: 
      [I 260528 15:00:00 server:335] Serving on http://127.0.0.1:8000
      Если перейти по адресу http://127.0.0.1:8000 в браузере, откроется рабочая версия сайта.

    Особенность mkdocs serve: Сервер поддерживает Live Reload (горячую перезагрузку). Это значит, что если изменить любой .md файл и нажать "Сохранить", страница в браузере обновится автоматически за долю секунды, без необходимости перезапускать сервер.

    Автоматизация развёртывания (GitHub Actions)¶

    Чтобы сайт автоматически обновлялся в интернете при каждом изменении файлов (при каждом git push), настроен пайплайн CI/CD в GitHub Actions.

    Скрипт лежит в файле .github/workflows/deploy.yml:

    name: Deploy MkDocs to GitHub Pages

on:
  push:
    branches:
      - main  # Запускать скрипт только при пуше в ветку main

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: 3.x

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Deploy to GitHub Pages
        run: mkdocs gh-deploy --force

    Как работает этот скрипт (объяснение процесса):¶

    1. Триггер (on: push): Как только разработчик отправляет новые коммиты в ветку main, GitHub запускает виртуальную машину (ubuntu-latest).
    2. Checkout: GitHub скачивает исходный код репозитория на эту виртуальную машину.
    3. Setup Python: Устанавливается язык Python.
    4. Установка зависимостей: Выполняется команда pip install -r requirements.txt. В файле requirements.txt у нас прописано: 
      mkdocs-material
      Эта команда скачивает MkDocs и выбранную тему.
    5. Сборка и Деплой (mkdocs gh-deploy): Эта команда заставляет MkDocs сгенерировать готовые HTML-страницы из Markdown и автоматически отправить их в специальную скрытую ветку gh-pages в этом же репозитории. GitHub Pages, в свою очередь, раздаёт файлы из этой ветки в интернет.

    Вывод¶

    В результате был создан удобный, современный и масштабируемый сайт-портфолио.

    Использование подхода Docs as Code (документация как код) позволяет версионировать отчёты вместе с исходным кодом программ, а настройка GitHub Actions полностью автоматизирует процесс доставки контента, исключая необходимость ручной сборки и выгрузки файлов.
